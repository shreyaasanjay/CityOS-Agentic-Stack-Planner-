# PlusCal Syntax Rules for TraceFix

Read this file before editing Protocol.tla. These rules are mandatory for the
PlusCal code inside the scaffold generated by compile_scaffold().

## The Big Rule

Every label must have an executable statement immediately after it. Do not stack
labels.

Wrong:

```tla
RESEARCHER_A_start:
rA_research:
  skip;
```

The first label has no statement, so pcal.trans may report an error such as:

```text
Expected ":=" but found "skip"
```

Correct:

```tla
RESEARCHER_A_start:
  skip; \* call: research_topic
rA_write:
  acquire_lock(DOCUMENT);
```

Correct loop form:

```tla
ra_loop: while (TRUE) {
  ra_research:
    skip; \* call: research_topic
  ra_submit:
    send(ch_sub_A, "submit");
};
```

## Closing Braces Need a Semicolon Before the Next Label

A bare `}` closing a `while`, `if`, or `either` block must be followed by `;`
before the next label can begin. A label directly after a bare `}` (no `;`)
produces a cryptic pcal.trans error like `Expected ";" but found "next_label"`.

Wrong:

```tla
ra_loop: while (TRUE) {
  ra_research:
    skip;
}
ra_done:
  skip;
```

Correct:

```tla
ra_loop: while (TRUE) {
  ra_research:
    skip;
};
ra_done:
  skip;
```

## Skip

`skip;` is valid PlusCal. It is the right no-op statement for domain work that
is represented only by a comment.

Never write this:

```tla
skip := TRUE;
```

That is an invalid assignment and usually means the real problem was an empty or
stacked label above the line.

## Editing Scaffold Placeholders

The scaffold starts each process with a placeholder like:

```tla
AGENT_start:
  skip; \* TODO: replace with AGENT's protocol logic
```

When replacing it, either keep `AGENT_start:` and put a real statement under it,
or replace the whole placeholder block with a sequence where every label has a
statement.

Wrong:

```tla
AGENT_start:
agent_work:
  skip;
```

Correct:

```tla
AGENT_start:
  skip; \* call: first_domain_tool
agent_work:
  acquire_lock(SHARED_RESOURCE);
```

## Statement Patterns

Use these forms:

```tla
label:
  statement;

loop_label: while (TRUE) {
  body_label:
    statement;
};

choice_label:
  either {
    branch_a:
      statement;
  } or {
    branch_b:
      statement;
  };

check_label:
  if (msg = "approve") {
    approve_label:
      statement;
  } else {
    reject_label:
      statement;
  };
```

Every statement and block ends with a semicolon.

## Macro Patterns

The scaffold defines these macros:

```tla
send(channel_id, "label");
receive(channel_id, msg);
acquire_lock(resource_id);
release_lock(resource_id);
acquire_counter(counter_id);
release_counter(counter_id);
```

Blocking operations need a label on or before them:

```tla
wait_for_editor:
  receive(editor_to_researcher, msg);
lock_document:
  acquire_lock(DOCUMENT);
```

## Comments and Domain Tools

Domain tool calls are comments after `skip;`. Do not put JSON objects or pseudo
tool calls in executable PlusCal syntax.

Correct:

```tla
ra_research:
  skip; \* call: research_topic {"topic": "subtopic A"}
```

Wrong:

```tla
ra_research:
  research_topic({"topic": "subtopic A"});
```

## Control Flow

- Labels are globally unique across all processes. Prefix labels with the agent
  name or initials.
- Execution falls through to the next label unless you use `goto`.
- A `goto` target must be a label in the same process.
- If a branch should return to a loop head, add an explicit `goto loop_label;`.
- A process reaches terminal state by falling through to its closing brace.

## Repair Checklist for PlusCal Syntax Errors

When pcal.trans reports a syntax error:

1. Read the reported line and at least 5 surrounding lines in Protocol.tla.
2. Look above the reported token for an empty label or two consecutive labels,
   or a bare `}` immediately followed by a label (missing `;`).
3. Do not change `skip;` into an assignment.
4. Make every label immediately own a statement or convert the label into a
   `while (TRUE)` loop label.
5. Re-run verify_spec().

## Editing Multiple Processes in One Turn

If you are about to call edit_file on a different process than the one you
just edited, call read_file('Protocol.tla') again first. An edit_file call
on Process A changes the line numbers and exact text of everything after it
in the file — the old_string you remember for Process B may no longer match
the current file even if Process B's own content didn't change. Do not chain
edit_file calls across multiple processes from a single earlier read_file.
