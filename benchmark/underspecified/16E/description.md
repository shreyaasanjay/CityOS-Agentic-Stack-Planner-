We're a small team of three engineers shipping a single service, and I'd like help pinning down our release routine. One of us writes the code: she takes our one build server, compiles the application, and runs unit tests on it, and once those pass she lets our tester know the build is ready. The tester needs that very same build server to run his integration tests, and only one person can be using the server at a time — so the developer has to be off it before the tester gets on, and the tester shouldn't start until he's actually heard the build is done. When integration tests pass, the tester tells our release engineer, who only then takes over our single deployment environment — nobody else touches it — pushes the application live, and runs a quick smoke test. After that passes, the release engineer tells both the developer and the tester that the deployment went out; each of them waits for that word before calling it a day. We're done when all three have wrapped up, strictly in that build, test, deploy order.
